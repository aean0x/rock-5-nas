# OpenClaw NixOS Module Extraction

Working document for extracting `hosts/system/openclaw/` into a standalone, reusable NixOS module.

## Design Principles

- Opinionated defaults, everything overridable
- Zero assumption about secret backend (sops, agenix, plain file)
- Workspace assembly engine is the core value — content is user-supplied
- Optional integrations for Caddy, OneDrive, etc.

## Target Option Surface

```nix
services.openclaw = {
  enable = true;
  domain = "openclaw.example.com";

  # Secrets — user provides path to dotenv file however they generate it
  secretsFile = "/run/openclaw.env";

  # Agent manifest — flake input or path containing agents.yaml
  agentsInput = inputs.openclaw-agents;

  # Image customization
  baseImage = "ghcr.io/phioranex/openclaw-docker:latest";
  extraDependencies = [];  # Typed dep list (same schema as packages.nix)

  # Workspace content (overridable per-document)
  workspace.documents = {
    "SOUL.md"   = { protected = "..."; initialPersistent = "..."; };
    "AGENTS.md" = { protected = "..."; initialPersistent = "..."; };
    "STYLE.md"  = { protected = "..."; initialPersistent = "..."; };
  };
  workspace.tasks = { };  # Lobster .lobster files

  # Agent tool policy
  agents.overrides = {
    # scout.grantPrivileged = [ "exec" "process" ];
    # coder.extraDeny = [ "browser" ];
  };
  agents.defaults.model.primary = "xai/grok-4.20-beta";

  # Gateway
  gateway.port = 18789;
  gateway.bind = "loopback";

  # Optional integrations
  integrations.caddy.enable = false;
  integrations.onedrive.enable = false;
};
```

## Extraction Checklist

### Phase 1: Decouple from host repo

- [ ] Replace `settings.domain` refs with `cfg.domain` module option
- [ ] Replace `settings.*` in config.nix (only domain used)
- [ ] Remove `services.openclaw.secretNames` plumbing (DONE - moved to dotenv)
- [ ] Replace `inputs.openclaw-agents` with `cfg.agentsInput` option
- [ ] Move sops template out of module — module only consumes `cfg.secretsFile` path

### Phase 2: Parameterize image build

- [ ] Expose `packages.nix` typed dependency list as a module option (`extraDependencies`) that merges with defaults
- [ ] Detect arch at eval time for tarball URLs (or accept aarch64-only for v1)
- [ ] Make base images configurable (`gatewayBaseImage`, `sandboxBaseImage`)
- [ ] `image.nix` type handlers and `mkStep` dispatcher stay as-is — already generic

### Phase 3: Externalize workspace content

- [ ] Define `workspace.documents` option type (attrset of { protected, initialPersistent })
- [ ] Ship skeleton defaults for each document (minimal, functional)
- [ ] Move current soul/style/agents content into the host repo's consumption of the module
- [ ] Keep the protected/persistent merge engine in the module (deployment.nix)

### Phase 4: Extract integrations

- [ ] Caddy: wrap in `mkIf cfg.integrations.caddy.enable` — expose port only by default
- [ ] OneDrive: move to `integrations.onedrive` sub-module, disabled by default
- [ ] Sops helper: optional `integrations.sops` that generates the template if sops-nix is present

### Phase 5: Packaging

- [ ] Create standalone flake with the module
- [ ] Add `flake.nixosModules.openclaw` export
- [ ] Document minimum viable config example
- [ ] Test with a clean NixOS config (not this repo)

## Known Pain Points

### sops-nix coupling
The dotenv file at `/run/openclaw.env` is generated by sops-nix templates outside the module.
Module just needs a path — user is responsible for populating it.
Optional sops helper can auto-generate the template if `integrations.sops.enable = true`.

### Workspace content is personal
Soul, style, agent rules are deeply user-specific. The module ships a minimal skeleton;
real content lives in the consuming repo. The merge engine (protected + persistent sections)
is the reusable part.

### Tarball URLs hardcode aarch64
`packages.nix` tarball entries (uv, Docker CLI, goplaces) use aarch64 URLs. Need to either:
- Detect `pkgs.system` and select URLs per-arch in `packages.nix`
- Use package managers that handle arch automatically
- Accept aarch64-only for now, document the limitation

### Caddy integration
`services.caddy.proxyServices` is a custom option, not upstream NixOS.
Module should just expose the port; reverse proxy wiring is the user's problem.
The optional Caddy integration is a convenience for users who have the same pattern.

### deployment.nix bash script
The marker-based document merge (sed/grep/cat) works but is the most fragile piece.
Consider: keep as-is for v1, potentially rewrite as a Nix derivation later.
The current approach has the advantage of preserving agent-written persistent sections
across rebuilds, which a pure derivation can't do (it would need a separate state mechanism).

## File Mapping (current -> module)

| Current | Module location | Notes |
|---|---|---|
| default.nix | module/default.nix | Entry point, options definition |
| config.nix | module/config.nix | Parameterize settings refs |
| agents.nix | module/agents.nix | Unchanged (already generic) |
| deployment.nix | module/deployment.nix | Core merge engine stays |
| packages.nix | module/packages.nix | Typed dep manifest, expose as option for merging |
| image.nix | module/image.nix | Type-dispatched builder, already generic |
| workspace/*.nix | module/workspace/ | Engine stays, content becomes defaults |
| onedrive.nix | module/integrations/onedrive.nix | Optional |
| AGENTS.md | stays in host repo | Documentation, not module code |
